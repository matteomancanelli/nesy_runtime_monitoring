Claude ha risposto:
Yes, essentially. But there's an important nuance about the "wait until I'm certain" part that's worth unpacking, because it's exactly what makes runtime monitoring distinct from offline trace checking.

The basic mechanism

You're right about the core idea. You compile the specification φ into a DFA A_φ once, offline. Then at runtime, as the trace is produced symbol by symbol, you maintain the current DFA state q. When a new symbol σ arrives, you advance: q' = δ(q, σ). This is constant time per step — just a table lookup. The DFA does all the heavy lifting that the formula's temporal structure requires; the monitor itself is essentially a one-state-pointer that walks the automaton.

The verdict question

This is where it gets interesting. The naive approach is: process the entire trace, then at the end check if the final state is in F. This works perfectly for offline trace checking, where you have the complete trace.

But in online runtime monitoring, traces are produced incrementally and may be very long (or even unbounded in the "finite but expanding" sense that RuleRunner emphasized). Waiting until the end is wasteful if you can already tell the verdict earlier. So the symbolic RV community developed the notion of good and bad prefixes:

A bad prefix is a finite prefix u such that no extension uv can satisfy φ. Once the monitor reaches a state from which no accepting state is reachable (a "trap" or "rejecting sink"), the verdict is definitively false — you can stop monitoring and report violation.
A good prefix is a finite prefix u such that every extension uv satisfies φ. Once the monitor reaches a state from which only accepting states are reachable (an "accepting sink"), the verdict is definitively true — you can stop monitoring and report success.
Otherwise the verdict is inconclusive (often denoted ?), meaning the trace could still go either way.
This is the three-valued semantics of LTL3 (Bauer, Leucker, Schallhart 2007). The four-valued RV-LTL refines this further by distinguishing "presumably true" (currently in an accepting state but could leave it) from "presumably false" (currently in a non-accepting state but could reach one). Both are built on top of the same DFA mechanism — they just classify states by their reachability properties.

Precomputing the verdict structure

The classical move in symbolic RV is to precompute, offline, a labeling of DFA states:

States from which no accepting state is reachable → label "definitely violates"
States from which all reachable states are accepting → label "definitely satisfies"
All other states → label "inconclusive"
This is done by standard graph reachability analysis on the DFA, done once after compilation. At runtime, advancing the state is still a single table lookup, and checking the verdict is also a single lookup of the precomputed label. So the per-step cost remains O(1), and you get early termination for free.

The LTLf-specific case

Here's where LTLf differs from classical LTL monitoring. In LTLf, the trace is complete when it ends — you're not monitoring a prefix of a potentially infinite execution, you're monitoring a finite, self-contained sequence. This means:

At the end of the trace, the verdict is always binary (true or false), determined by whether the final DFA state is in F.
During the trace, you can still apply the LTL3-style precomputation for early termination, but the "inconclusive" verdict means "we need to see more of this trace," not "we need to see what extends beyond it."
This is actually one of the cleaner aspects of working with LTLf: you don't have the philosophical complications of "what would happen if the trace continued forever" that plague classical LTL monitoring. Either the trace satisfies the formula at its end, or it doesn't.

What this means for your baseline

For your symbolic baseline using ltlf2dfa, the standard implementation pattern is:

Compile φ to DFA A_φ (offline, one-time cost).
Precompute trap states and accepting sinks (offline, one-time cost, done via two reachability checks on the DFA).
At runtime, for each trace symbol: advance the state, check if you've hit a sink, return early if so.
At trace end (if no early termination): check if final state ∈ F.
This is what tools like SPOT and the symbolic monitors built on ltlf2dfa do. It's also what the RuleRunner papers were comparing against, though they called the inconclusive verdict "undecided" and built their machinery around it.

The interesting comparison point

This early-termination capability is something you should track in your experiments. The three approaches handle it differently:

Symbolic DFA: early termination is essentially free (precomputed labels, one extra lookup per step).
RuleRunner: built-in via the three-valued semantics (the extended truth tables explicitly produce true/false/undecided verdicts per step).
DeepDFA: in its basic form gives a soft distribution over states; early termination requires either thresholding the distribution or maintaining the crisp DFA in parallel.
This is one place where DeepDFA, despite its advantages in differentiability and parallelism, might lose to the symbolic baseline on single-trace monitoring with frequent early termination. It's a fair point to make in the paper, and it sharpens the contribution: DeepDFA's advantage shows up in batched/learning settings, not in single-trace exact monitoring with early termination.

    def __init__(self, formula):
        self.dfa = ltlf2dfa(formula)        # compile
        self.trap_states = compute_traps()  # precompute
        self.accept_sinks = compute_sinks() # precompute
        self.state = self.dfa.initial
    
    def step(self, observation):

        if self.state in self.trap_states:
            return FALSE  # early termination
        if self.state in self.accept_sinks:
            return TRUE   # early termination
        return INCONCLUSIVE
    
    def final_verdict(self):
